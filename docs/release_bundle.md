# Release Bundle and Installer Design

This document describes the **current as-built** `.run` release bundle and installer behavior.

## Overview

The release artifact is a self-extracting installer named:

```text
redroid-modules-standalone-X.Y.Z.run
```

It is generated by `scripts/build-release-bundle.sh` and packages:

- workflow-built prebuilt modules under `prebuilt/<kernel-release>/`
- bundled `binder/` and `ashmem/` DKMS source trees
- `load_modules.sh`
- `scripts/verify-environment.sh` and related helpers
- `bin/ipcverify-host`
- `bin/ipcverify.apk`
- `VERSION` and `RELEASE_MANIFEST.txt`

## Bundle Layout

A simplified extracted layout looks like this:

```text
redroid-modules-standalone/
├── binder/
├── ashmem/
├── ipcverify/
├── scripts/
├── bin/
│   ├── ipcverify-host
│   └── ipcverify.apk
├── prebuilt/
│   └── <kernel-release>/
│       ├── binder_linux.ko
│       ├── ashmem_linux.ko
│       ├── KERNEL
│       ├── VERSION
│       └── SHA256SUMS
├── load_modules.sh
├── VERSION
└── RELEASE_MANIFEST.txt
```

For the full staged tree, also see `docs/RELEASE_BUNDLE_LAYOUT.md`.

## Install Flow

The installer logic lives in `packaging/run-installer/install.sh` and currently follows this sequence:

1. detect the running kernel with `uname -r`
2. look for `prebuilt/<kernel-release>/binder_linux.ko` and `ashmem_linux.ko`
3. install the matching prebuilt modules when present
4. fall back to bundled DKMS sources only when no exact prebuilt match exists and DKMS fallback is allowed
5. run `depmod`, load the modules, and mount `binderfs`
6. install `ipcverify-host` and stage the Android verifier artifacts
7. run `ipcverify-host --local-only --yes` unless `--skip-test` was requested

### Important policy

- the installer is **prebuilt-first**
- DKMS is a **fallback path**, not the primary install path
- `--no-dkms` disables that fallback entirely

## Release and Prebuilt Packaging Policy

Current release-prebuilt coverage is centered on:

- Ubuntu 24.04
- Debian 12 amd64
- Debian 13 amd64
- Proxmox VE 8
- Proxmox VE 9

RPM-family workflows (Fedora, Silverblue, RHEL / CentOS Stream, Amazon Linux 2, Amazon Linux 2023) currently validate DKMS builds in CI and collect logs/artifacts, even when those families are not all emitted as prebuilt release assets yet.

## Metadata and Integrity Files

The release bundle records the version and build metadata in multiple places:

- root `VERSION`
- per-kernel `prebuilt/<kernel-release>/VERSION`
- per-kernel `prebuilt/<kernel-release>/KERNEL`
- per-kernel `prebuilt/<kernel-release>/SHA256SUMS`
- root `RELEASE_MANIFEST.txt`

These files exist so the bundle stays deterministic and auditable.

## Operator Notes

- Use the `.run` installer for the normal path.
- Use `--extract-only` when you want to inspect or manually drive the extracted tree.
- If a kernel is not part of the prebuilt matrix, the bundled DKMS sources provide the fallback path.
- `ipcverify-host` is part of the normal post-install validation story and can also be run independently later.
